CSV Field Management Guide

How to add, remove, and sync fields between ACF, the PHP importer, and the PWA — without breaking anything.

How the system is wired

The CSV importer is defined in two places. Both need to match whenever you add or remove a field.

FileWhat it controls
class-vc-admin-importer.php WP admin import page + REST endpoint. Defines which columns are valid and how each one writes to ACF.
ImportEvents.jsx
ImportArtists.jsx, etc.		 PWA importer UI. Defines the column list for the preview table and template download. The CSV parser accepts any columns — this list governs what the user sees.
The short versionACF is the source of truth for what fields exist. The PHP class controls what the importer is allowed to write. The React component controls what the user sees. All three need to stay in sync.

Scenario 1 — You added a new field in ACF

Adding a new field to the CSV importer

You created a new field in ACF and want it to be importable via CSV. Three places to update:

1

Add it to get_columns() in PHP PHP

Open class-vc-admin-importer.php. Find get_columns() and add one line:

'new_field_key' => array( 'label' => 'Field label here' ),

If the field is required, add 'required' => true to the array.

2

Add write logic to apply_row() in PHP PHP

Find the apply_row() method. For a flat (non-group) field, add:

if ( $has( 'new_field_key' ) ) { update_field( 'new_field_key', sanitize_text_field( $data['new_field_key'] ), $post_id ); }

For fields inside an ACF group — use the merge pattern (see Scenario 3 below).

3

Add it to COLUMNS in React React

Open the relevant Import*.jsx file. Find the COLUMNS array at the top and add one entry:

{ key: 'new_field_key', label: 'New Field', notes: 'Short description' },

The preview table, template download, and column reference list all generate from this array automatically — no other React changes needed.

4

Deploy both files Deploy

PHP: SCP class-vc-admin-importer.php to the server.
PWA: git commit → git push → npm run build → copy build → push fc-bkstg-events.

Scenario 2 — Removing a field from the CSV

Removing a field from the importer

You no longer want a field to be importable — even if it still exists in ACF. Two removals, no ACF change needed.

1

Remove from PHP PHP

Delete the line from get_columns() and remove the matching $has() block in apply_row().

2

Remove from React React

Delete the entry from the COLUMNS array. If a user uploads a CSV that still has that column, it will be silently ignored.

Safe operationRemoving a field from the importer never deletes data from WordPress. It only stops that column from being written during future imports.

Scenario 3 — The field is inside an ACF group

Group fields — use the merge pattern

ACF groups like vc_ep_social, vc_ep_dates, and vc_ep_details require a read-merge-write pattern. Never call update_field() directly on a group subfield — ACF will blank out all other subfields in the group that weren't included.

// 1. Read the whole existing group $existing = get_field( 'vc_ep_social', $post_id ); $existing = is_array( $existing ) ? $existing : array(); // 2. Build only the values you want to write $updates = array(); if ( $has( 'new_subfield' ) ) { $updates['new_subfield'] = sanitize_text_field( $data['new_subfield'] ); } // 3. Merge and write the whole group back if ( ! empty( $updates ) ) { update_field( 'vc_ep_social', array_merge( $existing, $updates ), $post_id ); }
Why this mattersIf you write only the new subfield directly with update_field(), ACF blanks out all other subfields in the group. The merge pattern reads first, merges only your changes, then writes the full group — so untouched subfields survive.

Field type reference

ACF type → CSV format + PHP sanitizer

ACF TypeCSV Format & PHP Sanitizer
textPlain string — sanitize_text_field()
urlFull URL with https:// — esc_url_raw()
emailEmail address — sanitize_email()
numberPlain number, no commas — intval() or floatval()
selectOne of the allowed option values — sanitize_text_field() + validate against allowed list
true_false0 or 1 — (bool) intval()
date_pickerYYYY-MM-DD — sanitize_text_field() (ACF stores as Ymd internally)
wysiwyg / textarea⚠️ Not recommended — too long, formatting lost in CSV
image / file✖ Not supported — media must be uploaded separately
relationship✖ Not supported — requires post IDs/slugs plus lookup logic
taxonomySlug of the term — use wp_set_object_terms() instead of update_field()

Blank cells vs. missing columns

The most common source of accidental overwrites

SituationWhat happens
Column not in CSV at all ✓ Safe — field is completely ignored, existing value untouched
Column present, cell is blank ✓ Safe — the $has() check catches empty strings, existing value untouched
Column present, cell has a value ⚠️ Field is written with the new value, overwriting whatever was there
confidential or private column is 0 ⚠️ Explicitly sets the flag to false — this CAN un-confidential an event
Rule of thumbOnly include columns in your CSV that you actually want to update. Leave columns out entirely (not blank) for fields you want to leave alone. Blank cells are safe due to the $has() guard, but omitting the column is cleaner and less error-prone.

Add ACF field in WordPress (if it doesn't exist yet)
Add key to get_columns() in class-vc-admin-importer.php
Add write logic to apply_row() — use merge pattern if it's inside a group
Add entry to COLUMNS array in the relevant Import*.jsx file
SCP updated PHP file to server

Remove matching $has() block from apply_row()

SCP + deploy — existing WordPress data is untouched
